Memahami C4 Model Arsitektur

C4 Model adalah cara sistematis untuk menggambarkan arsitektur sistem dengan 4 level detail yang berbeda: Context, Container, Component, dan Code. Setiap level ditujukan untuk audiens yang berbeda, dari manajer bisnis hingga developer.

Apa itu C4 Model?

C4 Model adalah metodologi yang dikembangkan oleh Simon Brown untuk membuat dokumentasi arsitektur perangkat lunak yang mudah dipahami. Model ini menggunakan 4 level diagram yang semakin detail:

  • C1 - Context: Gambaran besar sistem dan penggunanya
  • C2 - Container: Aplikasi dan database yang membentuk sistem
  • C3 - Component: Komponen-komponen dalam setiap aplikasi
  • C4 - Code: Detail implementasi (opsional)

Mengapa Perlu C4 Model?

  • Komunikasi yang efektif: Setiap level untuk audiens yang tepat
  • Menghindari kebingungan: Tidak terlalu teknis atau terlalu abstrak
  • Dokumentasi yang hidup: Bisa diupdate bersama dengan kode
  • Onboarding yang cepat: Developer baru mudah memahami sistem

Prinsip Dasar C4

  • Hierarki: Mulai dari yang umum ke yang spesifik
  • Konsistensi: Gunakan notasi dan istilah yang sama
  • Konteks: Selalu jelaskan siapa yang menggunakan sistem
  • Simplicity: Fokus pada hal penting, abaikan detail teknis

4 Level C4 Model

Setiap level C4 memiliki tujuan dan audiens yang berbeda. Diagram di bawah menggunakan Mermaid untuk menunjukkan contoh sistem e-commerce sederhana.

C1 – System Context (Gambaran Besar)

Audiens: Manajer bisnis, stakeholder, investor
Tujuan: Menunjukkan sistem apa yang dibangun dan siapa yang menggunakannya

flowchart TB
actor((Pengguna)):::actor
extA([Sistem Pembayaran Eksternal]):::external
system[[Sistem E-Commerce]]:::system
actor -- Membeli barang --> system
system -- Konfirmasi pembayaran --> extA
classDef actor fill:#fff7ed,stroke:#fb923c,color:#7c2d12
classDef external fill:#eef2ff,stroke:#6366f1,color:#312e81
classDef system fill:#ecfeff,stroke:#06b6d4,color:#164e63

Pertanyaan yang dijawab: Apa yang dilakukan sistem? Siapa yang menggunakannya? Sistem eksternal apa yang berinteraksi?

C2 – Container (Aplikasi & Database)

Audiens: Developer, DevOps, arsitek teknis
Tujuan: Menunjukkan aplikasi dan database apa saja yang membentuk sistem

flowchart LR
subgraph Web[Web App]
  ui[Next.js Frontend]
end
subgraph API[API]
  svc[(REST API Service)]
end
subgraph Data[Data Layer]
  db[(PostgreSQL)]
  cache[(Redis Cache)]
end
mq[[Kafka]]
ui -->|HTTP/HTTPS| svc
svc -->|JDBC| db
svc -->|Cache| cache
svc --- mq

Pertanyaan yang dijawab: Aplikasi apa yang ada? Database apa yang digunakan? Bagaimana mereka berkomunikasi? Teknologi apa yang dipakai?

C3 – Component (Komponen dalam Aplikasi)

Audiens: Developer senior, arsitek perangkat lunak
Tujuan: Menunjukkan komponen-komponen utama dalam setiap aplikasi

flowchart TB
subgraph API Service
  controller[OrderController]
  service[OrderService]
  repo[OrderRepository]
  mapper[DTO/Entity Mapper]
end
controller --> service
service --> repo
service --> mapper
repo -->|SQL| DB[(PostgreSQL)]

Pertanyaan yang dijawab: Komponen apa saja dalam aplikasi? Bagaimana mereka berinteraksi? Apa tanggung jawab setiap komponen?

C4 – Code (Detail Implementasi)

Audiens: Developer yang akan mengerjakan kode
Tujuan: Menunjukkan detail implementasi seperti class diagram, sequence diagram

Level ini opsional dan biasanya dibuat hanya untuk area yang kompleks. Contohnya:

  • Diagram kelas untuk menunjukkan struktur kode
  • Sequence diagram untuk alur bisnis yang rumit
  • Database schema untuk tabel yang kompleks

Kapan dibuat: Hanya untuk bagian yang benar-benar kompleks dan perlu dokumentasi detail implementasi.

Tools untuk Membuat Diagram C4

Ada beberapa cara untuk membuat diagram C4, dari yang paling mudah hingga yang paling powerful. Pilih sesuai kebutuhan tim Anda.

1. Structurizr (Rekomendasi Utama)

Keunggulan: Dibuat khusus untuk C4, mendukung semua level, diagram-as-code, auto-layout

Contoh Structurizr DSL (C1–C3) 
// file: workspace.dsl
workspace "E-Commerce" "Contoh C4 dengan Structurizr" {
  model {
    // C1: Context
    user = person "Customer" "Pengguna yang membeli produk"
    system = softwareSystem "E-Commerce System" "Menjual produk online"
    user -> system "Membeli barang"
    
    // C2: Container
    web = container system "Web App" "Next.js" "UI untuk pelanggan"
    api = container system "API" "Spring Boot" "Business logic"
    db = container system "Database" "PostgreSQL" "Data transaksi"
    cache = container system "Cache" "Redis" "Caching data"
    
    user -> web "Mengakses UI"
    web -> api "HTTP"
    api -> db "JDBC"
    api -> cache "Cache"
    
    // C3: Component
    component api "OrderController" "HTTP Controller"
    component api "OrderService" "Business Logic"
    component api "OrderRepository" "Persistence"
  }
  
  views {
    systemContext system "Context" { include * autoLayout }
    container system "Container" { include * autoLayout }
    component api "API Components" { include * autoLayout }
  }
}
Install & Render (Windows PowerShell) 
# Install Scoop (jika belum ada)
Set-ExecutionPolicy RemoteSigned -scope CurrentUser
irm get.scoop.sh | iex

# Install Java dan Structurizr CLI
scoop install openjdk
scoop bucket add extras
scoop install structurizr-cli

# Render diagram ke PNG/SVG
structurizr-cli export -w workspace.dsl -o out -f plantuml -c workspace

2. C4-PlantUML (Familiar dengan PlantUML)

Keunggulan: Sintaks PlantUML yang familiar, library C4 yang lengkap, integrasi dengan banyak tools

Contoh C4-PlantUML 
@startuml
!includeurl https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Context.puml
!includeurl https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Container.puml

Container(api, "API", "Spring Boot", "Business logic")
ContainerDb(db, "Database", "PostgreSQL", "Data transaksi")

Rel(customer, web, "Mengakses UI")
Rel(web, api, "HTTP")
Rel(api, db, "JDBC")
@enduml
Install & Render (Windows PowerShell) 
# Install Java, Graphviz, dan PlantUML
scoop install openjdk graphviz plantuml

# Render file.puml ke PNG
plantuml -tpng -o out .\file.puml

3. Mermaid (Untuk Dokumentasi Ringan)

Keunggulan: Mudah dipelajari, terintegrasi dengan GitHub/GitLab, tidak perlu library khusus C4

Catatan: Mermaid tidak memiliki tipe diagram C4 resmi, tapi bisa digunakan untuk merepresentasikan ide C4 dengan flowchart/graph.

Contoh Mermaid untuk C2

  end
  subgraph API[API Layer]
    SVC[Spring Boot API]
  end
  subgraph Data[Data Layer]
    DB[(PostgreSQL)]
    CACHE[(Redis)]
  end
  
  UI -->|HTTP| SVC
  SVC -->|JDBC| DB
  SVC -->|Cache| CACHE
Install Mermaid CLI (Windows) 
# Install Node.js dan Mermaid CLI
scoop install nodejs-lts
npm i -g @mermaid-js/mermaid-cli

# Render file.mmd ke PNG

Langkah-langkah Membuat Diagram C4

Ikuti workflow ini untuk membuat dokumentasi arsitektur yang efektif dengan C4 Model.

1. Mulai dari C1 - System Context

Identifikasi sistem utama dan semua pihak yang berinteraksi dengannya.

  • Siapa pengguna sistem? (customer, admin, developer)
  • Sistem eksternal apa yang berinteraksi? (payment gateway, email service)
  • Apa tujuan utama sistem?
  • Fitur utama apa yang disediakan?

2. Lanjut ke C2 - Container

Pecah sistem menjadi aplikasi dan database yang membentuknya.

  • Aplikasi apa saja yang ada? (web app, mobile app, API)
  • Database apa yang digunakan? (PostgreSQL, MongoDB, Redis)
  • Bagaimana mereka berkomunikasi? (HTTP, gRPC, message queue)
  • Teknologi apa yang dipakai? (Next.js, Spring Boot, Docker)

3. Turun ke C3 - Component

Untuk setiap aplikasi, identifikasi komponen-komponen utamanya.

  • Controller/API layer (menangani request HTTP)
  • Service layer (business logic)
  • Repository layer (akses data)
  • Utility/Helper classes

4. Opsional C4 - Code

Hanya untuk area yang benar-benar kompleks dan perlu dokumentasi detail.

  • Class diagram untuk struktur kode yang rumit
  • Sequence diagram untuk alur bisnis yang kompleks
  • Database schema untuk tabel yang rumit

5. Simpan sebagai Diagram-as-Code

Simpan file diagram di repository dan setup auto-render.

  • Simpan file .dsl/.puml/.mmd di folder docs/
  • Setup CI/CD untuk auto-render diagram
  • Update diagram setiap kali ada perubahan arsitektur
  • Gunakan version control untuk tracking perubahan

Tips Sukses

  • Mulai sederhana: Jangan over-engineering di awal
  • Update rutin: Diagram harus selalu up-to-date dengan kode
  • Kolaborasi: Libatkan tim dalam review diagram
  • Fokus pada yang penting: Abaikan detail teknis yang tidak relevan

Butuh Bantuan Membuat Diagram C4?

Berikan deskripsi singkat sistem Anda (fitur utama, pengguna, teknologi), saya bisa bantu membuatkan diagram C1–C3 yang siap digunakan dengan Structurizr DSL atau PlantUML.

Dibuat untuk memperkenalkan C4 Model. Diagram Mermaid di halaman ini adalah representasi sederhana untuk memudahkan pemahaman.
